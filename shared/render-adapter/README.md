# @celebratebanner/render-adapter

The last translation step before pixels. It connects the **Render Orchestrator** to
the **existing renderer** — the orchestrator decides, the adapter translates, the
renderer draws:

```
Memory Profile → Creative Brief → WOW Engine → Render Orchestrator
    → Render Adapter → Renderer → Rendered Preview Images → Premium Reveal
```

The adapter makes **no creative decisions**. It does not change layouts, renderer
algorithms, photo placement, or typography. It only carries a `RenderPlan`'s
already-decided values into a renderer-ready request, drives the renderer, and
packages the result. It embeds **no pixels**: photos travel as references and
rendered outputs travel as opaque image handles (a `uri`), never as base64/dataURL.

## Public API

```ts
renderConcept(renderPlan: RenderPlan, renderer: Renderer, options?): RenderedConcept
```

- `renderPlan` — the accepted plan from `@celebratebanner/render-orchestrator`.
- `renderer` — an **injected** `Renderer` port (see below). The adapter never imports
  or modifies the production render engine; it only calls `renderer.render(request)`.
- `options` — `{ bannerText?, seed?, previewLongEdge?, thumbnailLongEdge?, renderExports? }`.

A plan the orchestrator did **not** accept is never rendered: `renderConcept` returns
a `skipped` result and does not touch the renderer.

### The `Renderer` port

```ts
interface Renderer {
  render(request: RenderRequest): RenderedImage;
}
```

The adapter hands the renderer a fully-translated `RenderRequest` (arrangement, hero +
supporting photo references, palette, typography, background, spacing, layering, seed,
target dimensions, color mode) and takes back a `RenderedImage` handle. Binding the
production engine is a thin wrapper the caller owns — for example:

```ts
import { renderPreview, renderHD } from '@celebratebanner/render-engine';

const renderer: Renderer = {
  render(req) {
    const canvas = makeCanvas(req.widthPx, req.heightPx); // env-specific
    const input = toEngineInput(req, resolveImages(req));  // map refs → images
    (req.kind === 'export' ? renderHD : renderPreview)(canvas, input);
    return { /* targetId, kind, format, size, uri, byteSize, renderMs */ };
  },
};
```

The tests inject a deterministic, pixel-free stub renderer instead (see
`test/stub-renderer.ts`), which is also how the snapshot fixtures stay reproducible.

## `RenderedConcept`

| Field | Meaning |
|-------|---------|
| `conceptName` | the concept that was rendered |
| `previewImage` | fast on-screen preview image handle (RGB png) |
| `thumbnailImage` | small thumbnail image handle (RGB png) |
| `renderStatus` | `completed` \| `skipped` (plan not accepted) \| `failed` |
| `renderTime` | total ms — sum of every image's reported `renderMs` |
| `exportTargets` | one rendered image per export target (digital + three print) |
| `qualityChecks` | render completed · preview exists · thumbnail exists · export targets available · upstream plan passed |

(plus `schemaVersion`, `occasion`, `arrangement`.)

## Quality gate

`renderConcept` accepts a result only when **all** of these hold:

- render completed
- preview exists
- thumbnail exists
- all four export targets were produced
- the upstream `RenderPlan` was itself accepted

Otherwise `qualityChecks.passed` is `false` and `reasons[]` explains why.

## Determinism

The adapter uses no `Date`, `Math.random`, or I/O. Given a deterministic renderer it
produces byte-identical output — the committed `fixtures/*.json` are `RenderedConcept`
snapshots generated by running the full pipeline through the stub renderer.
